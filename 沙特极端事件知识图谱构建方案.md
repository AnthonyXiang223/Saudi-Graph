# 沙特极端事件知识图谱构建方案

## MAZU 多灾种早期预警智能体 — 知识图谱子系统

---

## 一、项目背景

本项目紧扣中国气象局 MAZU 系列早期预警系统"多灾种、预警、零差距、普惠"核心定位，以沙特阿拉伯为目标合作国家，针对热带沙漠气候下突发性暴雨、极端高温干旱等高频高影响气象灾害，构建轻量化、智能化、可嵌入 MAZU 系统的多灾种早期预警知识图谱。

**数据基础**：全球大气再分析（ERA5）、地表实况、海表温度（OSTIA）、卫星云图（GPM IMERG）等多源气象数据，覆盖 2025 年全年（365 天），空间范围 16.0°N–31.9°N, 34.0°E–55.9°E，~0.1° 分辨率，共 35,200 个格点。

---

## 二、架构设计原则

知识图谱采用**知识层与数据层分离**的架构：

```
┌──────────────────────────────────────────────────┐
│ 知识层 (networkx DiGraph — 只管知识关系)            │
│                                                  │
│  DataSource ──► Indicator ◄── derived_from ──►   │
│                    │  │                          │
│                    │  co_occurs_with              │
│                    ▼                             │
│  Rule ──► HazardType ◄── Event ──► Region        │
│                                                  │
│  节点量级：111 节点 + 428 边                        │
├──────────────────────────────────────────────────┤
│ 数据层 (xarray — 只管时空栅格)                      │
│                                                  │
│  NetCDF → xr.open_dataset().load() → LRU cache   │
│  35,200 格点 × 365 天 × ~100 变量                  │
└──────────────────────────────────────────────────┘
```

**核心原则**：
- **KG 不做**：不为每个格点建节点、不为每天建节点、不建空间邻接边、不建时间序列边
- **KG 只做**：维护指标→算子→规则→灾害→事件之间的知识关系网
- **桥接方式**：Indicator.id = NetCDF variable name，数据查询全部下沉到 xarray

## 三、知识图谱本体设计

### 3.1 节点类型（5 类）

| 节点类型 | 数量 | 说明 |
|---|---|---|
| `Indicator` | **91** | 每个极端事件指标一个节点，携带元数据、算子 DAG、可用性 |
| `DataSource` | **6** | DS1(ERA5月值)、DS2(ERA5日值)、DS4(ERA5极值)、DS8(GHCN气候态)、DS10(GPM卫星降水)、SST(OSTIA海温) |
| `HazardType` | **4** | flash_flood(山洪)、extreme_heat(极端高温)、dust_storm(沙尘强风)、coastal_humid_heat(沿海湿热) |
| `Region` | **6** | saudi_bbox、red_sea、persian_gulf、north_saudi、central_saudi、south_saudi |
| `Rule` | **4** | 每种灾害一条加权检测规则，含 primary gate 和 fallback 策略 |

### 3.2 关系类型（7 类）

| 关系 | 方向 | 含义 | 边数 |
|---|---|---|---|
| `derived_from` | Indicator → Indicator | 算子依赖链 | 42 |
| `co_occurs_with` | Indicator ↔ Indicator | 物理上需联合解释（双向） | 264 |
| `sourced_from` | Indicator → DataSource | 原始数据来源 | 101 |
| `contributes_to` | Indicator → HazardType | 指标对该灾害有判别贡献 | 17 |
| `detects` | Rule → HazardType | 规则用于检测该类灾害 | 4 |
| `instance_of` | Event → HazardType | 事件类型归属 | 动态 |
| `located_in` | Event → Region | 事件空间归属 | 动态 |

### 3.3 图统计

| 指标 | 值 |
|---|---|
| 总节点数 | 111 |
| 总边数 | 428 |
| 平均度数 | ~7.7 |
| derived_from 子图 | 有向无环（DAG）✓ |

## 四、指标算子体系

### 4.1 九大类 91 个指标

| 类别 | 数量 | 数据源 | 说明 |
|---|---|---|---|
| 月尺度地表/能量 | 14 | DS1 | 月降水、辐射、热通量、风应力 |
| 云量 | 4 | DS2 | 总/低/中/高云量 |
| 日降水/辐射/能量 | 10 | DS2 | 日降水、对流降水、净辐射、热胁迫 |
| 近地面热湿/风 | 14 | DS2+DS4 | 气温、露点、湿度、VPD、热指数、风速 |
| DS8气候态距平/热浪 | 12 | DS8+DS2/DS4 | 降水距平、温度距平、热浪标志/持续天数 |
| 对流稳定度 | 6 | DS2 | CAPE、CIN、抬升指数、地形 |
| 多层动力/水汽输送 | 20 | DS2 | IVT、风切变、涡度、散度、垂直速度 |
| SST/卫星降水 | 10 | SST+DS10 | 海温、GPM短历时降水、交叉验证 |
| 综合风险 | 1 | DS2+DS10 | flash_flood_risk (0-5评分) |

### 4.2 算子编码格式

每个指标算子包含三个层次的可执行表达：

```json
{
  "id": "t2m_anomaly_c",
  "description": "2米气温距平，正值表示偏暖",
  "expression": "t2m_c - t2m_climatology_c",
  "dag": {"op": "sub", "left": {"op": "var", "name": "t2m_c"}, "right": {"op": "var", "name": "t2m_climatology_c"}},
  "inputs": ["t2m_c", "t2m_climatology_c"],
  "output_unit": "degC",
  "source": "DS2+DS8",
  "co_occurs_with": ["t2m_c", "tmax_anomaly_c", "heatwave_day_flag"],
  "availability": {"effective_days": 362, "total_days": 365}
}
```

| 字段 | 消费者 | 用途 |
|---|---|---|
| `description` | LLM / Agent | 自然语言理解 |
| `expression` | 人类开发者 | 文档查阅 |
| `dag` | DAG 解释器 | 可执行计算 |

### 4.3 DAG 操作符词汇表

`kg/operators.py` 内置 OPS 词汇表，包含 16 个基础操作符：

**算术**：`add`, `sub`, `mul`, `div`, `neg`
**幂与根**：`sqrt`, `sqr`, `pow`
**聚合**：`max`, `min`, `abs`
**逻辑/条件**：`threshold`, `where`, `and_`, `or_`
**变量引用**：`var`

外加气象专用操作符（`vpd_formula`, `heat_index_formula`, `sum_flags` 等），所有 `operators.json` 中的 DAG 只允许使用已注册操作符，加载时自动校验。

### 4.4 关键算子链示例

```
DS2.t2m(K) ──[-273.15]──► t2m_c ──┐
DS8.TMAX/MDAY ──[nn]──► tmax_climatology_c ──┤
                                              ├──► tmax_anomaly_c
                                              │
tmax_c ──[>=max(40, climatology+5)]──► heatwave_day_flag
                                          │
              ──[consecutive count]──► heatwave_duration_days
```

## 五、业务规则体系

### 5.1 算子与规则分离

| 文件 | 内容 | 稳定性 |
|---|---|---|
| `schema/operators.json` | 物理公式、DAG、指标元数据 | 不变（物理定律） |
| `schema/rules.json` | 检测阈值、权重、严重度分级、fallback | 可调（业务需求） |

### 5.2 四类灾害检测规则

| 灾害 | 条件数 | Primary Gate | Fallback | 连通域 |
|---|---|---|---|---|
| **山洪** | 5 | flash_flood_risk ≥ 3 | ds10_max_1h 缺失 → 降权重 | 4-连通, ≥3格点 |
| **极端高温** | 4 | heatwave_day_flag ≥ 1 | 无 | 4-连通, ≥5格点 |
| **沙尘强风** | 4 | wind10_speed ≥ 12 m/s | wind_shear_850_200 缺失 → 降权重 | 8-连通, ≥10格点 |
| **沿海湿热** | 4 | sst_celsius ≥ 30°C | 无 | 4-连通, ≥3格点 |

### 5.3 Primary Gate 机制

每个规则设有一个主条件（primary gate）：只有主条件满足的格点，风险分才被完整保留；主条件未满足的格点，风险分压制 75%（×0.25）。

### 5.4 连通域分析

使用 `scipy.ndimage.label` 对阈值化后的风险矩阵做连通域标记，每个连通域独立生成一个 Event 节点。事件属性包括：受影响格点数、面积（km²）、质心坐标、所属地理分区。

## 六、检测引擎架构

### 6.1 与已有 compute_indicators.py 的分工

| 层 | 负责方 | 输入 | 输出 |
|---|---|---|---|
| **基础层** | `compute_indicators.py`（已有） | DS2/DS10 原始变量 | `flash_flood_risk` (0-5简单计数) 等预计算指标 |
| **升级层** | `event_detector.py`（新增） | 基础层全部指标 | 加权风险分 + 连通域 + Event节点 + 严重度分级 |

### 6.2 运行时流程

```
输入: "2025-08-19", hazard_type="flash_flood"
  │
  ├──► 1. KG 查: flash_flood → Rule → conditions
  ├──► 2. 检查 availability → fallback strategy
  ├──► 3. datalayer.load("20250819") → Dataset (LRU cached)
  ├──► 4. 逐条件: xarray.where(indicator >= threshold) → bool masks
  ├──► 5. 加权求和 → primary gate 压制 → 归一化
  ├──► 6. scipy.ndimage.label(mask, connectivity) → 连通域簇
  ├──► 7. 过滤 min_cluster_size → 每个簇生成 Event 节点
  └──► 8. 返回 + 挂接 KG
```

## 七、查询接口

三层查询架构：

```python
# 知识层（纯图遍历，不碰数据）
kg.get_hazard_indicators("flash_flood")
kg.get_indicator_chain("heatwave_duration_days")
kg.get_operator("vpd_kpa")

# 数据层（xarray 操作）
kg.data.get_timeseries(lat=24.7, lon=46.7, start, end, variables)
kg.data.get_spatial_snapshot("2025-08-19", "daily_precip_total")

# 联合层（KG 驱动 + 数据执行 + 可解释输出）
kg.detect_events("2025-08-19", ["flash_flood"])
kg.explain(event)  # 自然语言，逐条件审计
```

## 九、项目文件结构

```
f:\Saudi\
├── schema/                              # 知识层定义
│   ├── ontology.json                    # 本体：5类节点 + 7类关系
│   ├── operators.json                   # 91个指标算子（含DAG）
│   └── rules.json                       # 4类灾害检测规则
├── kg/                                  # 知识图谱引擎
│   ├── ontology.py                      # networkx 图构建 + 验证 + 查询
│   ├── operators.py                     # OPS 词汇表（16操作符）+ DAG 解释器
│   ├── datalayer.py                     # xarray 封装 + LRU 缓存 (maxsize=7)
│   ├── event_detector.py                # 加权评分 + 连通域 + primary gate + fallback
│   ├── query.py                         # 三层查询入口 + CLI
│   └── scan_availability.py             # 全量 365 天覆盖扫描
├── dashboard/                           # Web 可视化界面
│   ├── server.py                        # Flask 后端 API
│   ├── static/vis-network.min.js        # 图谱渲染库（本地）
│   └── templates/index.html             # 交互式前端
├── indicators/                          # 365 个 NetCDF 日文件
├── verify.py                            # 5 项集成验证脚本
└── saudi_extreme_event_indicator_report.md  # 指标计算原始报告
```
