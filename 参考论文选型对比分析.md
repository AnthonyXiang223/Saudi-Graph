# 沙特极端气候预警知识图谱 — 参考论文选型对比分析

> 对比四篇相关论文，评估与当前项目的差异，给出 Agent 开发的架构选型建议。

---

## 一、四篇论文概览

| 论文 | 发表 | 核心方法 | 数据类型 | 灾种 | 规模 |
|---|---|---|---|---|---|
| **GeoAI** (Tian & Li) | 2022, LIPIcs | 3规则级联因果抽取 (主题→时空约束→文本挖掘) | NWS 风暴文本报告 | 飓风/洪水/暴雨 | 3K节点, 17.8K边 |
| **KGFPO** (Du et al.) | 2025, Env. Mod. & Soft. | OWL本体 + 全过程观测建模 | 传感器/卫星/UAV观测资源元数据 | 洪水 | 郑州7.20+新乡7.21案例 |
| **Drought KG** (Fang et al.) | 2023, J. Hydroinf. | 本体设计 + BERT-BiLSTM-CRF NER | CNKI中文文献文本 | 干旱 | 文献语料库 |
| **Agro-Meteo ST-KG** (Zhang et al.) | 2023, Remote Sensing | OWL+GeoSPARQL+SWRLTO时空推理 | MODIS NDVI + ESA气象再分析 + 农田矢量 | 干热风 (dry hot wind) | 河南3县, 农田斑块级 |

## 二、逐篇深度对比

### 2.1 GeoAI：级联事件因果知识图谱

**做什么**：从 NWS（美国国家气象局）风暴文本报告中自动抽取飓风→洪水→暴雨的因果链。三阶段流水线：

```
阶段1: 主题关联 → 阶段2: 时空共现约束 → 阶段3: 文本挖掘因果词
         ↓                    ↓                        ↓
   查询扩展匹配          7天时间缓冲 +             "lead to", "result in"
   飓风/洪水/暴雨       空间重叠过滤              等因果短语模式匹配
```

**关键技术指标**：

| 方法 | 精确率 | 召回率 | F1 |
|---|---|---|---|
| 纯模式匹配 | 100% | 4% | — |
| BERT 分类 | 76% | 71% | — |
| 混合 | — | — | 74.23% |

- 从 62,151 句 → 20,000 个因果对 → 聚类后 3,000 节点 + 17,801 边
- 聚类前 15,686 个碎片子图 → 聚类后 1 个连通大图

**与我们项目的差异**：

| 维度 | GeoAI | 我们的项目 |
|---|---|---|
| **知识来源** | 非结构化文本报告 | 结构化 NetCDF 数值指标 |
| **因果类型** | 事件→事件（飓风导致洪水） | 指标→灾害（CAPE≥1000 贡献于山洪） |
| **时空粒度** | 事件级别（粗粒度） | 格点级别 0.1°×逐日（细粒度） |
| **KG 构建方式** | NLP 文本抽取 → 规则过滤 | 专家定义的算子 JSON → networkx 建图 |
| **推理能力** | 因果链查询（A 导致 B？） | 阈值条件 + 加权评分 + 连通域检测 |
| **自动化程度** | 全自动 (文本→因果对) | 专家手动定义算子 + 自动执行 |
| **可解释性** | 抽取文本片段作为因果证据 | DAG + primary gate + 逐条件审计 |

**可借鉴之处**：
- ⭐⭐⭐ **因果关系建模思路**：他们的"事件→下游事件"逻辑可以延伸为我们的"指标异常→灾害事件→次生灾害"链
- ⭐⭐ **混合方法（规则+ML）**：我们当前的阈值规则（100%精确，覆盖有限）可以后续用 ML 补充（提高召回，类似他们的 BERT 分支）
- ⭐ 时空共现约束：我们已有 implicit（连通域分析），但可以显式标注级联关系

### 2.2 KGFPO：洪水全过程观测知识图谱

**做什么**：构建 OFPO（洪水过程观测本体），用 OWL 建模洪水全生命周期——减灾→备灾→响应→恢复。核心创新是把**观测资源**（卫星/无人机/地面传感器/模型方法）也编入图谱，实现"任务需要什么数据→哪个传感器能提供→用什么模型分析"的自动关联。

**本体结构**：

```
OFPO 核心概念层级:
├── FloodEvent (洪水事件)
│   ├── time, location, severity
│   └── hasObservationTask
├── ObservationTask (观测任务)
│   ├── requiresData
│   └── usesMethod
├── DisasterData (灾害数据)
│   ├── satellite imagery
│   ├── meteorological data
│   └── in-situ measurements
├── ModelingMethod (建模方法)
│   ├── hydrological model
│   └── ML prediction
└── ObservationResource (观测资源)
    ├── Satellite (高分系列, Sentinel...)
    ├── UAV
    └── GroundSensor
```

**与我们项目的差异**：

| 维度 | KGFPO | 我们的项目 |
|---|---|---|
| **关注点** | 观测资源配置与管理 | 极端事件检测与预警 |
| **建模对象** | Task-Data-Method-Sensor 关联 | Indicator-DataSource-Hazard 关联 |
| **形式化** | OWL + 推理规则 + 三元组存储 | networkx + JSON schema + Python 规则引擎 |
| **数据** | 观测资源元数据（传感器类型/时空覆盖） | 数值网格数据（35,200格点 × 365天 × ~100变量） |
| **推理** | SWRL 规则推理观测资源适配 | Primary gate + 加权评分 + 连通域 |
| **生命周期** | 全过程（减灾→备灾→响应→恢复） | 当前聚焦"预警"阶段 |

**可借鉴之处**：
- ⭐⭐⭐ **全过程建模思路**（高价值）：我们的项目目前只覆盖"预警"阶段。KGFPO 的全生命周期模型可以指导我们扩展 KG 到灾前、灾后阶段——比如加 `MitigationTask`、`RecoveryAssessment` 节点
- ⭐⭐ **OWL 形式化**：如果未来需要与 MAZU 或其他国际系统对接，OWL + SPARQL 是标准方案。当前 networkx 适合原型，但跨系统互操作需要 OWL
- ⭐ **观测资源建模**：当前项目中 DataSource 只是 metadata。如果 MAZU 系统需要根据预警等级自动调度卫星/无人机观测，可以借鉴 OFPO 的 Task→Resource 关联

### 2.3 Drought KG：基于本体+NER的干旱知识图谱

**做什么**：从 CNKI 中文干旱文献中，用 BERT-BiLSTM-CRF 模型自动抽取干旱相关的实体和关系，构建干旱知识图谱。分两步：(1) 手动设计干旱本体；(2) 用深度学习 NER 从大量文献中自动抽取实体填充图谱。

**技术栈**：本体建模（OWL-like）→ 语料标注 → BERT-BiLSTM-CRF 训练 → NER 抽取 → 知识融合 → 图谱构建

**与我们项目的差异**：

| 维度 | Drought KG | 我们的项目 |
|---|---|---|
| **知识来源** | 非结构化科学文献文本 | 结构化气象数值数据 |
| **构建方式** | 深度学习 NER 自动抽取 | 专家手动编码 91 个算子 |
| **实体类型** | 地名、灾种、指标、作物…（从文本中识别） | 预定义的 5 类节点 |
| **关系抽取** | NER 识别后规则/模型抽取 | operators.json 的 inputs/co_occurs_with 字段 |
| **适用场景** | 知识发现（从海量文献总结干旱规律） | 实时预警（在每日数据上运行） |

**可借鉴之处**：
- ⭐⭐ **文献知识自动化抽取**：我们的 operator description 字段目前是手动写的。如果未来需要从论文/报告中自动扩展指标体系，NER + 关系抽取可以加速
- ⭐⭐ **本体 + 数据双驱动**：他们先设计本体（专家知识），再用 NER 填充（数据驱动）。我们当前是纯专家驱动（91个算子全部手写），后续可以加入数据驱动来发现新的指标关联
- ⭐ 命名实体识别思路：对沙特本地化的阿拉伯语气象报告做 NER，提取本地事件描述

### 2.4 Agro-Meteo ST-KG：农业气象灾害时空知识图谱

**做什么**：这是**最接近我们项目**的一篇。针对河南冬小麦干热风灾害，构建时空知识图谱，融合多源异构数据（MODIS NDVI 遥感 + ESA 气象再分析 + 农田矢量数据），用 OWL + GeoSPARQL + SWRLTO 做时空推理，实现了农田斑块级别的灾害监测。

**系统架构**：

```
数据层:
  MODIS MOD09GA (500m, 每日 NDVI)
  + ESA 气象再分析 (1km, 每小时: T2m, 露点, 风 u/v)
  + 农田斑块矢量 (GLASS LAI 作物分布)
      ↓
语义层:
  OWL 本体 (灾害事件 + 农田 + 气象要素 + 遥感指数)
  + GeoSPARQL (空间查询)
  + SWRL Temporal Ontology (时间推理)
      ↓
推理层:
  SWRL 规则: IF 温度≥30°C AND 相对湿度≤30% AND 风速≥3m/s
              FOR 连续2天
              THEN 干热风预警
  GeoSPARQL: 查询"哪些麦田斑块落入预警区域"
      ↓
输出:
  农田斑块级预警 (vs 传统县级粗粒度)
  NDVI 下降验证灾害强度
```

**关键技术决策**：

| 决策 | 方案 | 原因 |
|---|---|---|
| 本体语言 | OWL | 国际标准，跨系统互操作 |
| 空间语义 | GeoSPARQL (OGC 标准) | 支持空间拓扑查询（相交、包含、缓冲） |
| 时间语义 | SWRLTO | 支持事件持续时长、前后顺序推理 |
| 遥感融合 | NDVI 直接纳入推理规则 | 数值阈值触发 + 植被变化验证 |
| 粒度 | 农田斑块 (500m) | 比传统县级预警精细 100× |

**与我们项目的差异（关键）**：

| 维度 | Agro-Meteo ST-KG | 我们的项目 |
|---|---|---|
| **数据融合深度** | 遥感 (NDVI) + 气象 (T/RH/wind) 联合推理 | 气象为主，SST 仅作辅助背景 |
| **空间粒度** | 农田斑块 (500m 像素) | 0.1° 格点 (~11km) |
| **形式化程度** | OWL + GeoSPARQL + SWRL | JSON schema + networkx |
| **时间推理** | SWRLTO 规则：连续2天满足条件 | heatwave_duration_days（预计算连续计数） |
| **推理规则** | SWRL 形式化：IF-THEN 语义规则 | Python 函数：加权评分 + 阈值 |
| **输出承灾体** | 叠加农田斑块 → 田块级预警 | 当前仅格点 → Region，无承灾体叠加 |
| **验证方式** | NDVI 下降验证灾害真实影响 | 报告统计 + 反例验证 |

**与我们项目的相似之处（最重要）**：

| 相似点 | 他们 | 我们 |
|---|---|---|
| 数据驱动指标 | ESA 气象 → T2m/RH/wind | ERA5 → tmax_c/rh2m/wind10_speed |
| 阈值条件检测 | 温度≥30°C AND 湿度≤30% | heatwave_day_flag: tmax≥max(40, clim+5) |
| 多源融合 | 遥感 + 气象 + 农田 | DS2 + DS4 + DS8 + DS10 + SST |
| 时空结构化 | OWL + GeoSPARQL + SWRLTO | xarray (格点) + networkx (知识) 分离 |
| 灾害类型 | 干热风（高温+低湿+风） | 极端高温 + 沙尘强风 + 沿海湿热 + 山洪 |

**可借鉴之处（⭐⭐⭐ 最高优先级）**：

1. **承灾体叠加**（最重要）：他们把 NDVI + 农田斑块叠加到预警格点上，实现了"哪些田块受威胁"。我们可以把沙特的城市、关键基础设施、港口、农田叠加到检测出的 Event 上
2. **GeoSPARQL 空间查询**：如果 MAZU 需要支持"查询红海沿岸 50km 内所有山洪事件"，GeoSPARQL 是标准方案
3. **SWRL 推理规则形式化**：我们的 rules.json 目前是自定义 JSON 格式。如果用 SWRL 标准，可以对接更多语义推理工具
4. **遥感 NDVI 验证机制**：他们用 NDVI 下降验证干热风影响。我们可以用沙特的 MODIS/Landsat 植被指数验证高温干旱的实际影响
5. **农田斑块级粒度**：我们当前 ~11km 格点对沙特沙漠区域可能够用，但对红海沿岸城市洪涝需要更高分辨率

## 三、综合对比矩阵

| 维度 | GeoAI | KGFPO | Drought KG | Agro-Meteo ST-KG | **我们的项目** |
|---|---|---|---|---|---|
| KG 构建方式 | NLP 文本抽取 | OWL 本体工程 | 本体 + NER | OWL + GeoSPARQL + SWRL | JSON schema + networkx |
| 数据驱动 vs 专家驱动 | 数据驱动 | 专家驱动 | 混合 | 混合 | 纯专家驱动 |
| 时空粒度 | 事件级 | 观测资源级 | 文献文档级 | 农田斑块级 (500m) | 格点级 (~11km) |
| 数值计算 | 无 | 无 | 无 | SWRL 规则阈值 | DAG 解释器 + 加权评分 |
| 因果推理 | 事件→事件 级联 | Task→Data→Method | NER 抽取的关系 | IF-THEN 规则 | derived_from + contributes_to |
| 形式化标准 | 自定义语义规则 | OWL | OWL-like | OWL + OGC 标准 | 自定义 JSON |
| 承灾体 | 无 | 无 | 作物/地区 | 农田斑块 | Region (只有地理分区) |
| 实时性 | 离线分析 | 离线分析 | 离线分析 | 准实时 | 每日数据可实时检测 |
| 可解释性 | 文本片段证据 | 推理规则溯源 | NER 抽取来源 | SWRL 规则链 | DAG + primary gate + explain() |

## 四、针对当前项目的选型建议

### 4.1 我是否选对了技术路线？

**是的，核心路线正确，但需要补充三个关键能力**。

**做对了的**：
- ✅ 知识层/数据层分离 → 比四篇论文都更清晰
- ✅ DAG 可执行算子 → 四篇都没有这个（最独特的优势）
- ✅ Primary gate + fallback → 针对数据缺测的工程化处理
- ✅ explain() 可解释性 → 在 GeoAI 的文本证据思路上更进了一步

**需要补充的**（按优先级）：

### 4.2 优先级 P0：承灾体叠加（借鉴 Agro-Meteo ST-KG）

**当前**：Event 只输出格点坐标和面积，不知道伤了谁。

**借鉴 Agro-Meteo ST-KG 的农田斑块思路**：在 KG 里新增 `Asset`（承灾体）节点类型：

```
Asset 节点示例:
├── Riyadh_city     (城市: 人口 7M, 关键基础设施)
├── Jeddah_port     (港口: 红海主要港)
├── Dammam_refinery (工业设施: 炼油厂)
├── wheat_region_A  (农业区: 灌溉麦田)
└── highway_40      (交通: 利雅得-达曼高速)

关系:
  Event ──affects──→ Asset (从 Event 的连通域质心 + 缓冲半径判断)
  Asset ──located_in──→ Region
```

实现方式：GeoSPARQL 或 scipy 空间索引 + 简单的 `point_in_polygon`。

### 4.3 优先级 P1：OWL 形式化（借鉴 KGFPO）

**当前**：全部自定义 JSON/YAML → networkx。适合原型，但跨系统互操作差。

**建议**：保留当前 JSON schema 作为"源格式"，增加一个 `kg/to_owl.py` 转换器，生成 OWL 文件，用于：
- 对接 MAZU 系统中的其他语义组件
- 支持 SPARQL 复杂查询（如"哪些指标在两小时内同时触发阈值"）
- 学术报告和论文中引用标准化的本体定义

```
当前 pipeline:
  operators.json → ontology.py → networkx

增加后:
  operators.json → ontology.py → networkx (原型/内部)
                               → to_owl.py → OWL 文件 (MAZU 对接/标准查询)
```

### 4.4 优先级 P2：级联事件因果链（借鉴 GeoAI）

**当前**：事件检测是单时间片运行，不跟踪事件演化。

**建议**：在连续日期的 Event 之间建立 `caused` / `preceded` / `intensified` 关系：

```
2025-08-19 红海暴雨 ──caused──→ 2025-08-19 山洪
2025-08-19 山洪     ──intensified→ 2025-08-20 滑坡（如有数据）
2025-08-19 山洪     ──preceded──→ 2025-08-20 洪水漫溢
```

实现方式：相邻日期的 Event 用空间重叠（IoU）+ 时间连续性自动关联。这是 GeoAI 的"级联事件"思路在我们的数值系统上的延伸。

### 4.5 优先级 P3：数据驱动知识发现（借鉴 Drought KG 的 NER 思路）

**当前**：91 个算子全部手动编写。

**建议**：不改变当前专家驱动模式，但补一条数据驱动 pipeline：

```
每日 NetCDF → 相关性分析 (哪些指标总是同时异常？)
                   ↓
            发现新的 co_occurs_with 候选
                   ↓
            专家审核 → 加入 operators.json
```

这不是替代专家，是用数据辅助专家发现潜在的关联。

### 4.6 不需要借鉴的

| 论文 | 不需要的部分 | 原因 |
|---|---|---|
| GeoAI | 全文本 NLP pipeline | 我们有结构化 NetCDF，不是非结构化文本 |
| KGFPO | 观测资源调度 | MAZU 系统不涉及传感器调度（至少当前阶段） |
| Drought KG | 文献 NER | 我们没有海量文献需要自动抽取 |
| Agro-Meteo ST-KG | SWRLTO 时间推理 | 我们已有的 heatwave_duration_days 更简单直接 |

## 五、总结

| 优先级 | 借鉴来源 | 做什么 |
|---|---|---|
| **P0** | Agro-Meteo ST-KG | 承灾体叠加：Asset 节点 + affects 关系 |
| **P1** | KGFPO | OWL 形式化：kg/to_owl.py 转换器 |
| **P2** | GeoAI | 级联事件：相邻日 Event 的时序因果链 |
| **P3** | Drought KG | 数据驱动发现：自动挖掘 co_occurs_with 候选 |
| — | 全部四篇 | 它们都是离线分析型 KG，而我们已经是准实时+可执行 DAG。这是我们的项目最大的差异化优势 |

**一句话**：我们的 **DAG 可执行算子 + 知识/数据分层 + explain() 可解释性** 是四篇论文都没做到的独特优势。P0 承灾体叠加是最能快速提升项目价值的补充。
